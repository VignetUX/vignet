❯ What I want is test assertions driving a storybook-like workshop, where if the user changes parameters it still renders but doesn't run 
   the assertions. In other words, leveraging existing unit tests rather than requiring a developer to define another set of usually      
  quite similar logic in storybook to render components. To do this, I want to be able to filter tests (e.g., via an annotation or        
  decorator), run tests in isolation within an iframe rather than in batch, and parameterize some test inputs that can be changed in the  
  workshop. Do you think it's worth leveraging vitest-related components in this case?                                                    
                                                                                                                                          
⏺ This is a well-defined use case. Let me give you a direct architectural take.                                                           
                                                                                          
  The core problem with using vitest's runner                                                                                             
                                                                                                                                          
  The fundamental issue is that vitest doesn't model "render phase" vs "assertion phase" — a test body is just a function. To support     
  "re-run with new params but skip assertions," you need to intercept or replace the execution layer. That's not something vitest exposes 
  as a hook; you'd be fighting its batch-oriented runner.                                                                                 
                                                        
  The other problem: parameter injection. If you want the developer to write a normal-ish test that accepts workshop parameters, you need 
  some injection point. There's no natural one in vitest's execution model.
                                                                                                                                          
  What's genuinely worth keeping from vitest                                                                                              
   
  Keep:                                                                                                                                   
  - Vite dev server + the browser package's Vite plugins — specifically the module transform pipeline that makes vi.mock() work
  in-browser. If any of the tests use mocking, you need this. If they don't, plain Vite is sufficient.                                    
  - vitest's collect() API — scanning test files to find annotated tests without running them. This is already built and handles
  describe/it nesting, file parsing, etc.                                                                                                 
  - @vitest/expect — the assertion library itself, importable standalone. In workshop mode you'd wrap it in a no-op proxy; in test mode   
  you use it normally.                                                                                                                 
  - The iframe-per-test isolation model — valid, but you can implement this yourself in ~50 lines. You don't need the orchestrator        
  machinery.                                            
                                                                                                                                          
  Don't bother with:                                    
  - The orchestrator/tester as-is — replace entirely with your workshop UI                                                                
  - The provider abstraction — for an interactive developer tool, the browser is already open. Playwright/WebdriverIO is for automated    
  headless runs; a workshop is the opposite.                                                                                          
  - The RPC protocol — design a simpler one for your actual data model (component state, param values, render output)                     
  - The batch runner — it's the wrong execution model                                                                
                                                                                                                                          
  The custom execution layer you'd build                                                                                                  
                                                                                                                                          
  The key insight is that you only need a thin custom it wrapper that captures test bodies rather than executing them immediately:        
                                                        
  // workshop-runtime.js (runs inside the iframe)                                                                                         
  const registry = []                                   
                                                                                                                                          
  globalThis.it = globalThis.test = (name, fn) => {
    registry.push({ name, fn })                                                                                                           
  }                                                                                                                                       
   
  // After the test module loads, you have all test bodies captured.                                                                      
  // To render in workshop mode:                        
  async function renderWorkshop(testIndex, params) {                                                                                      
    const { fn } = registry[testIndex]                                                                                                    
    const mockExpect = new Proxy(expect, {
      apply: () => noopMatcher  // assertions become no-ops                                                                               
    })                                                                                                                                    
    globalThis.expect = mockExpect                                                                                                        
    await fn() // runs the render, skips assertions                                                                                       
    globalThis.expect = expect // restore for real test runs                                                                              
  }                                                                                                                                       
                                                                                                                                          
  For parameter injection, the cleanest API that doesn't require rewriting existing tests:                                                
                                                        
  // Developer writes:                                                                                                                    
  import { param } from '@your-workshop/runtime'                                                                                          
   
  it('renders a button', () => {                                                                                                          
    const label = param('label', 'Click me') // in workshop mode: reads from controls; in test mode: returns default
    render(<Button label={label} />)                                                                                                      
    expect(screen.getByRole('button', { name: label })).toBeVisible()
  })                                                                                                                                      
                                                        
  param() reads from a workshop-injected global in workshop mode, returns the default in normal test execution. No test logic changes     
  required.                                             
                                                                                                                                          
  For the annotation/filter:                                                                                                              
   
  it.workshop('renders a button', () => { ... })                                                                                          
  // or just a file comment: // @workshop                                                                                                 
  // or a config glob: workshopTests: ['**/*.workshop.test.ts']
                                                                                                                                          
  The real architecture                                                                                                                   
                                                                                                                                          
  Your workshop (Vite app, custom UI)                                                                                                     
    ↓                                                                                                                                     
  Vite dev server (with vitest's browser plugins if mocking needed)
    ↓                                                                                                                                     
  iframe per selected test                              
    ↓                                                                                                                                     
  Custom workshop-runtime.js inside iframe              
    - intercepts it/describe/expect                                                                                                       
    - registers test bodies
    - exposes renderWorkshop(index, params) to parent                                                                                     
    - communicates via postMessage (simple, no RPC needed)                                                                                
    ↓                                                                                                                                     
  Component renders in iframe DOM                                                                                                         
                                                                                                                                          
  The workshop UI talks to iframes via postMessage — send params, receive "render complete" or errors. No WebSocket, no vitest RPC needed.
   
  Bottom line                                                                                                                             
                                                        
  For your exact use case, vitest is worth using as a source of Vite plugins and the test collection API, not as the runtime. The         
  @vitest/browser package's Vite plugin pipeline is the hardest part to replicate (especially module mocking), so if your target tests use
   vi.mock(), lean on it. If they don't mock, plain Vite gets you 90% of the way there with far less complexity.                          
                                                        
  The orchestrator, tester, providers, RPC — all of that is the wrong abstraction for an interactive workshop. You'd end up with cleaner  
  code writing your own thin iframe coordinator than trying to adapt vitest's batch-run infrastructure.
